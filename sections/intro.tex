\section{Introduction}
\label{sec:intro}

A/B tests on tech platforms run in the thousands per day. The classical
fixed-horizon $t$-test assumes the sample size is chosen before looking
at the data. In practice analysts peek. They stop experiments that
look significant or extend those that look close, which can
inflate the false-positive
rate~\cite{johari2017peeking}. The same
\emph{optional stopping} problem arises in clinical trials, where
interim analyses are essential but classical group-sequential designs
require the number of looks to be fixed in advance. Anytime-valid
confidence sequences~\cite{howard2021time,
waudbysmith2024estimating, whitehouse2025timeuniform} solve this:
they provide confidence intervals valid at every stopping time.
The non-asymptotic flavour (Howard-Ramdas, betting, Whitehouse vector)
gives hard guarantees at every $t$. The asymptotic flavour
(time-uniform CLT) gives the large-sample behaviour that governs
deployment-scale runtime.

The theory is clean. The code that computes the confidence band is not.
A missing absolute value, an off-by-one on a log, or a dropped
symmetry hypothesis can each invalidate the coverage guarantee while
the software still appears to run. A Lean~4 proof pinned against a
fixed Mathlib commit removes that class of error. Writing these proofs
by hand is labour-intensive: each theorem in the
companion library required multiple iterations of tactic
refinement against a changing Mathlib API. Language-model drafters could reduce this effort, but
existing Lean-proof benchmarks
(MiniF2F~\cite{zheng2022minif2f},
ProofNet~\cite{azerbayev2023proofnet}, Verina~\cite{sunblazeverina})
measure closure against frozen statement libraries. They do not test
whether a drafter can navigate a library with its own conceptual
structure: custom definitions, multi-module import chains, and
domain-specific lemma vocabulary. Which drafter capabilities are
required to close theorems about such a structure, and can those
capabilities be decomposed into prompt-level interventions?

To answer this we introduce \textsc{Formal-AVS}, a benchmark of
$60$ Lean~4 theorems on anytime-valid confidence sequences across
four families. The targets range from single-function sign lemmas to
cross-function inequalities that require composing rate definitions
from all four families simultaneously. A companion Lean library
(\texttt{formal-avs-lean},\footnote{\url{https://anonymous.4open.science/r/neurips-2026-dataset-paper-replication-2604/supplementary/lean/}} Apache-2.0, ${\sim}3$k lines)
supplies the rate functions Mathlib does not yet provide.
The benchmark is paired with a $14$-session Harmonic
Aristotle~\cite{aristotle2025} history archive containing closed
proofs, counterexample witnesses, and axiom audits. Prior Lean
benchmarks release target statements or solver traces.
\textsc{Formal-AVS} releases both.

We evaluate four closed-source generalists, two open-weights
provers (DSPv2-7B~\cite{dspv22025},
Goedel-Prover-V2-8B~\cite{goedelprover2025}), and Harmonic
Aristotle across three capability levels (single-shot, agentic,
unbounded). Single-shot generalist closure peaks at $22/60$
(Gemini, hint-list prompt) and drops to zero on cross-function
targets. A four-round agentic loop
(\texttt{cycle\_prove} $r{=}4$~\cite{opengauss2025}) closes
the first non-Aristotle cross-function target under agentic mode and narrows
the prompt-sensitivity gap: Gemini's hint-list advantage
vanishes ($19 = 19$) under agentic mode, suggesting the
refinement loop partially substitutes for prompt engineering.
Aristotle, under an unbounded compute budget, closes $53/60$
axiom-clean and surfaces three false-as-stated targets that
all other solvers accepted
(\S\ref{sec:aristotle-refute}). We also report a prompt-sensitivity
matrix and a T3 hint ablation
(Table~\ref{tab:prompt-sensitivity}).

The benchmark is organized into six tiers. T0--T3 are defined
by proof shape before any solver is run. T4 and T5 are empirical
failure categories assigned after evaluation. Tier details
and a representative target are in~\S\ref{sec:methods-tiers}
and Appendix~\ref{app:bench-format-fig}.
